Server Specifications
ข้อกำหนด Server (Server Specification Analysis)
ข้อกำหนด Hardware ที่แนะนำสำหรับแต่ละสภาพแวดล้อม พร้อมเหตุผลทางเทคนิค
เหตุผลที่ต้องการทรัพยากรสูง (Why High Specs Matter)
  • 1

    AI/OCR Processing — CPU Intensive

    ประมวลผลรูปภาพด้วย Sharp + ส่งให้ OpenRouter API แบบ concurrent หลาย request พร้อมกัน ใช้ CPU สูงมากช่วง Batch Upload

  • 2

    File Upload สูงสุด 10 GB ต่อไฟล์

    MAX_UPLOAD_SIZE_MB=10240 — ไฟล์ขนาดใหญ่ต้องการ Memory Buffer และ I/O Speed สูง

  • 3

    PostgreSQL Shared Instance — 4 Databases

    PostgreSQL รองรับ 4 databases พร้อมกัน ต้องการ RAM สูงสำหรับ shared_buffers (25% RAM)

  • 4

    Excel/VLookup Processing — Memory Heavy

    ExcelJS อ่านไฟล์ Excel ทั้ง Workbook ขึ้น Memory ก่อน ต้องการ RAM เพียงพอสำหรับ Hash Map ใน VLookup Engine

  • 5

    12+ Docker Containers บน Host เดียว

    Production รัน 12+ containers พร้อมกัน แต่ละ Node.js process ใช้ RAM ~200–500 MB baseline

Security Checklist (Production)
  • ✓

    Rate Limiting

    15 req/15 min บน Auth endpoint — ป้องกัน Brute-force

  • ✓

    JWT + Refresh Token Rotation

    BlacklistedToken table ป้องกัน session hijack

  • ✓

    Full Audit Trail

    AuditLog, FileActivityLog, AdminActionLog — ทุก action บันทึก

  • !

    SSL/TLS Certificate (แนะนำ)

    ควรตั้ง HTTPS ด้วย Let's Encrypt บน nginx reverse proxy

  • !

    PostgreSQL Network Isolation (แนะนำ)

    ควร bind postgres เฉพาะ Docker internal network ไม่ expose port 5433 ออก internet

RS256 Auth & Security — ระบบความปลอดภัย

Authentication Flow
graph TD Login["User Login\n(Username + Password)"] --> Check{"Rate Limit\nRedis: 5 attempts / 60s"} Check -->|"ยังไม่ถึง Limit"| Verify["ตรวจสอบ Credentials"] Check -->|"เกิน Limit"| Block["429 Too Many Requests\n(Block 60 วินาที)"] Verify -->|"สำเร็จ"| Sign["Sign JWT\n(RS256 Private Key)"] Sign --> Token["Return JWT Token"] Request["API Request + JWT"] --> Pub["Verify Signature\n(RS256 Public Key)"] Pub -->|"Valid"| Allow["✓ Grant Access"] Pub -->|"Invalid / Expired"| Deny["401 Unauthorized"]
RS256 vs HS256
HS256Shared Secret Key — ทั้งสองฝั่งใช้ key เดียวกัน ถ้า key หลุดทั้งคู่เสี่ยง
RS256Asymmetric Keys — Private Key สร้าง JWT, Public Key ตรวจสอบ เฉพาะ API ที่ถือ Private Key เท่านั้นออก Token ได้
Security Features
  • ✓

    RS256 Asymmetric JWT

    Private key เก็บเฉพาะ Server ที่ออก Token เท่านั้น

  • ✓

    Redis Rate-Limiting Login

    จำกัด 5 attempts / 60 seconds บน Login endpoint — Counter ใน Redis พร้อม TTL 60s

  • ✓

    Redis Dual-use

    Redis ใช้ทั้ง BullMQ Job Queue และ Rate-limiting Counter บน Instance เดียวกัน
